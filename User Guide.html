What is OopisOS?

OopisOS (v2.0) is a fully persistent, self-contained operating system simulation that runs entirely within your web browser. Born from a collaborative effort between Andrew Edmark and Gemini Pro, it aims to recreate the feel and functionality of a classic command-line interface (CLI) environment, complete with a hierarchical file system, user management with passwords, a powerful text editor, a suite of useful commands, an AI interaction layer, and even an interactive text adventure game engine.

It's an educational tool, a nostalgic playground, and a testament to what can be achieved with JavaScript, IndexedDB for persistence, and a lot of creative coding. The entire system is now modularized across OopisOSv2_0.html and its accompanying JavaScript files.

Core Philosophy

  • Persistence is Key: Your files, user accounts (with hashed passwords), and session states are saved using IndexedDB and localStorage, meaning your work remains even after closing the browser tab.
  • Self-Contained: Everything runs client-side. No server needed (beyond serving the initial HTML/JS).
  • Educational & Fun: Designed to be a learning experience for those interested in OS concepts, CLI interactions, and JavaScript module architecture. Includes AI interaction and gaming!
  • Extensible: Built with modularity in mind, making it possible for developers to contribute new commands and features.
  • POSIX-ish Familiarity: While not a strict POSIX implementation, many commands and concepts will feel familiar to users of Unix-like systems.

Key Features at a Glance (v2.0)

  • Persistent File System: Create, edit, move, copy, and delete files and directories. Stored in IndexedDB. Commands include `mkdir`, `ls` (with flags like `-l`, `-a`, `-R`, `-t`, `-S`, `-d`), `cd`, `pwd`, `touch` (with `-c`, `-d`, `-t`), `cat`, `cp` (with `-r`, `-f`, `-p`, `-i`), `mv` (with `-f`, `-i`), `rm` (with `-r`, `-f`), `tree` (with `-L`, `-d`), `find` (with predicates like `-name`, `-type`, `-mtime`, `-user`, `-perm` and actions like `-print`, `-exec`, `-delete`).
  • Advanced Command-Line Interface: Retro-styled terminal with history, tab completion, command pipelining (`|`), I/O redirection (`>`, `>>`), background processes (`&`), and command sequences (`;`).
  • User Management with Passwords: Register multiple users (`useradd`/`register`) with secure password prompts, log in/out (`login`, `logout`), switch user (`su`, including to `root`), `whoami`, `listusers`, `removeuser` (with confirmation). Each user has an isolated home directory. Includes default users `Guest` (no password) and `userDiag` (password: `pantload`).
  • Enhanced Text Editor (`edit`):** Supports plain text, Markdown, and HTML (with live preview). Features line numbers (gutter hides on word wrap), toggleable word wrap (persists), status bar, view modes (`Ctrl+P`), `Ctrl+S` (save/exit), `Ctrl+O` (exit/confirm discard via modal), a Markdown/HTML formatting toolbar (with shortcuts like `Ctrl+B` for bold, `Ctrl+I` for italic), and an HTML preview export function.
  • AI Integration (`gemini`): Send prompts and file content to a Gemini AI model directly from the terminal. Can be used with a file path and a prompt, or with just a prompt string.
  • Scripting Engine (`run`): Execute script files with comments (`#`), shebang support (ignored for interpreter, but line is skipped), and argument passing (`$1`, `$@`, `$#`). Includes `delay` for pauses and `check_fail` for basic test scripting.
  • Session Management: Automatic session saving, manual `savestate`/`loadstate` (with FS snapshot and confirmation), `backup`/`restore` via JSON (with confirmation), `clearfs` (user's home directory wipe with confirmation), and full `reset` (all data wipe with confirmation).
  • Utilities: `echo`, `date`, `help`, `clear`, `history -c`, `export` (download files from VFS), `upload` (upload files to VFS, with type checks and overwrite prompt), `chmod`, `chown`, `grep` (with `-i`, `-v`, `-n`, `-c`, `-R` for recursive search), and `printscreen ` (saves current terminal output to a specified file in the VFS).
  • Interactive Text Adventure Game (`adventure`): Launch and play text-based adventure games, such as the included 'Quest for the Lost Semicolon,' within a dedicated modal window. Supports loading custom adventures from JSON files stored in the VFS.

Dive in, explore, and see what you can create in the world of OopisOS v2.0!

The File System

OopisOS has a hierarchical file system, much like Linux or macOS.

  • The root directory is /.
  • Paths are separated by /.
  • . refers to the current directory.
  • .. refers to the parent directory.
  • Hidden files and directories start with a . (e.g., .config) and are not shown by ls unless -a is used.

You can navigate using cd, list contents with ls, and create directories with mkdir.

The Editor (`edit` command)

OopisOS includes a powerful text editor accessible via the edit command. It's designed for both simple text editing and authoring Markdown or HTML with live preview capabilities.

  • Modes: Supports plain text, Markdown, and HTML. The mode is automatically detected based on the file extension (e.g., .md for Markdown, .html or .htm for HTML).
  • Live Preview: For Markdown and HTML files, a preview pane can be toggled to see rendered output in real-time.
  • Formatting Toolbar: When editing Markdown or HTML files, a toolbar appears with buttons for common formatting actions like bold, italic, lists, code blocks, etc.
  • Word Wrap: Can be toggled on/off. This setting is saved and persists across sessions. When word wrap is off, horizontal scrolling is enabled for the textarea. The line number gutter is hidden when word wrap is active.
  • Status Bar: Displays information like current line/column, total lines, word count, and character count.
  • Export Preview: For Markdown or HTML files, the rendered preview can be exported and downloaded as an HTML file.
  • Keyboard Shortcuts:
    • Ctrl+S: Save the current file and exit the editor.
    • Ctrl+O: Exit the editor. If there are unsaved changes, a modal dialog will prompt for confirmation to discard them.
    • Ctrl+P: Toggle the preview mode (cycles through Split View, Edit Only, Preview Only for Markdown/HTML files).
    • Ctrl+B: Apply/remove bold formatting (in Markdown/HTML modes).
    • Ctrl+I: Apply/remove italic formatting (in Markdown/HTML modes).
    • Tab: Inserts spaces for indentation.

Users, Permissions, and Passwords

OopisOS supports multiple users, each with their own home directory and saved session state. Each file and directory has an owner and permissions that determine who can read, write, or execute it.

User Accounts and The Superuser (`root`)

  • Default Users: OopisOS starts with three default users:
    • Guest: The default user, has no password.
    • root: The superuser account, has no password by default. The `root` user can bypass all file permission checks, allowing for system-wide administrative tasks like deleting other users' files.
    • userDiag: A special user for testing purposes, pre-configured with the password `pantload`.
  • register [username] (or useradd): Creates a new user account. You will be interactively prompted to set a password. If you press Enter at the password prompt, the user will be created without one.
  • login [username] [password]: Logs in as an existing user. If the user requires a password and you don't provide one, you will be prompted securely. The `login` command also accepts the password as a second argument for non-interactive scripting.
  • logout: Logs out the current user and returns to the Guest session.
  • su [username]: Switches the current user. If no username is given, it defaults to `root`. If the target user has a password, you will be prompted for it.
  • whoami: See your current username.
  • listusers: Show all registered user accounts.
  • removeuser [username]: Deletes a user and all their associated data (files, saved sessions). This action requires confirmation.

Permissions and Ownership

  • chmod [mode] [path]: Change the permissions of a file or directory. The mode is specified as two octal digits (e.g., 75 for rwxr-x, 64 for rw-r--). The first digit applies to the owner, the second to others. Each digit is a sum of: 4 (read), 2 (write), 1 (execute).
  • chown [new_owner] [path]: Change the owner of a file or directory. Only the current owner of the item or the `root` user can change its ownership.
  • Permissions are displayed in the long listing format of the ls -l command (e.g., -rwxr-x--- where 'd' indicates a directory, followed by owner and other permissions).
Guest@OopisOs:/> login userDiag pantload
Logged in as userDiag.
userDiag@OopisOs:/home/userDiag> _

Pipes, Redirection, and Command Sequences

Like other shells, OopisOS supports features to control command input, output, and execution flow:

  • Pipes |: Send the output of one command as input to another.
    Example: ls -a / | grep docs
  • Output Redirection >: Write command output to a file, overwriting it if it exists.
    Example: echo "My important data" > data.txt
  • Append Output >>: Append command output to a file.
    Example: date >> /system.log
  • Command Separator ;: Run multiple commands in sequence on the same line.
    Example: mkdir new_dir ; cd new_dir ; echo "I'm here" > file.txt

OopisOS v2.0: Advanced Topics & Features

Ready to delve deeper? This section explores some of the more advanced capabilities and concepts within OopisOS v2.0.

Scripting with `run`

OopisOS supports shell scripting via the run command (defined in commexec2_0.js). Create text files (e.g., myscript.sh using the edit command) containing a sequence of OopisOS commands, one per line.


# Example: /scripts/myscript_v2_0.sh
#!/bin/oopis_shell # Shebang line is read but interpreter isn't used; it's for convention.
echo "Starting my OopisOS v2.0 script..."
# This is a comment
mkdir /test_data_v2_0
echo "Hello from script v2.0, argument 1 was: $1" > /test_data_v2_0/script_output.txt
ls /test_data_v2_0
echo "All arguments: $@"
echo "Number of arguments: $#"
echo "Script finished."

To execute (assuming it's saved as /scripts/myscript_v2_0.sh and made executable):

Guest@OopisOs:/> chmod 70 /scripts/myscript_v2_0.sh
Guest@OopisOs:/> run /scripts/myscript_v2_0.sh firstArg "second arg with spaces"

Script Features:

  • Comments: Lines starting with # (outside of quotes) are ignored.
  • Shebang: Lines starting with #! are recognized and the line is skipped; the specified interpreter is not actually used.
  • Argument Passing:
    • $1, $2, ...: Access positional arguments passed to the script.
    • $@: Represents all arguments passed to the script, as a single string with spaces between them.
    • $#: Represents the total number of arguments passed to the script.
  • Error Handling: If any command in the script fails (returns `success: false`), the script will terminate at that point.
  • Utilities for Scripting:
    • delay [milliseconds]: Pauses script execution.
    • check_fail "[command_string]": Executes the provided command string. If the command string fails, check_fail succeeds. If the command string succeeds, check_fail fails (halting the script). Useful for automated testing.

Text Adventure Gaming (`adventure`)

OopisOS v2.0 features a built-in text adventure game engine, allowing users to embark on interactive fiction journeys. This functionality is primarily managed by modules in adventure2_0.js.

  • Launching: Type adventure to launch the default sample game, 'Quest for the Lost Semicolon.'
  • Custom Adventures: You can load custom adventures by providing a path to a JSON file within the OopisOS VFS: adventure /path/to/my_game.json.
  • Interface: The game plays out in a dedicated modal window, separate from the main terminal, offering an immersive experience with styled output for room names, descriptions, items, etc.
  • Engine: The TextAdventureEngine handles command parsing (e.g., look, go north, take key), game state (player location, inventory), and interactions with the game world defined in the JSON data.

Background Processes (`&`)

You can run a command or a pipeline asynchronously in the background by appending an ampersand (&) to the end of the command line. The `CommandExecutor` handles this, allowing you to continue using the terminal while the background task completes. OopisOS will assign a job ID to the backgrounded process and notify you when it starts and finishes (or if it encounters an error).

Guest@OopisOs:/> delay 5000 &
[1] Backgrounded.
Guest@OopisOs:/> echo "I can still type!"
I can still type!
(After approximately 5 seconds, a message like "[Job 1 finished]" will appear)

Recursive Grep (`grep -R`)

The grep command in v2.0 supports the -R flag (or --recursive) for recursively searching through directories. This powerful option allows you to find patterns within files across an entire directory tree, not just in specified files or standard input.

Guest@OopisOs:/projects> grep -R "TODO" .
./projectA/main.js: // TODO: Refactor this complex function
./projectB/docs/readme.txt: - TODO: Add examples for advanced usage
... (and so on for all matches containing "TODO" in the current directory and its subdirectories)

OopisOS v2.0: Developer Documentation

Welcome, aspiring OopisOS contributor or module developer! This section provides insights into the codebase and how you can extend OopisOS v2.0.

Code Structure Overview

The primary logic for OopisOS is now split into four main JavaScript files, all referenced by index.html:

  • oopisos2_0.js: Contains the core OS infrastructure, including configuration, general utilities, and managers for Output, Storage (LocalStorage), IndexedDB, FileSystem, History, Confirmation, User accounts (with Passwords), Sessions, Terminal UI, and Tab Completion.
  • editor2_0.js: Houses all logic related to the built-in text editor, including its specific configuration, utility functions, UI management, core manager, and modal dialogs.
  • commexec2_0.js: Dedicated to command execution. It includes the `CommandExecutor` module which contains the `Lexer`, `Parser`, and the definitions and handlers for all built-in commands.
  • adventure2_0.js: Contains all modules related to the text adventure game feature, including the game engine, modal interface, and the sample adventure data.

Each file generally uses Immediately Invoked Function Expressions (IIFEs) to create modules, encapsulating logic and avoiding global namespace pollution. Key managers and utilities are exposed for inter-module use where necessary (e.g., `FileSystemManager` being used by commands in `commexec2_0.js`).

Contributing to OopisOS

The spirit of open source and learning is encouraged!

  1. Understand the Code: Familiarize yourself with the modules in their respective files.
    • For general OS behavior, core APIs, and persistence: oopisos2_0.js.
    • For command-related work (adding/modifying commands, parsing): commexec2_0.js.
    • For editor enhancements: editor2_0.js.
    • For the adventure game feature: adventure2_0.js.
  2. Identify an Area: Bug fix, command improvement, new command, editor feature, adventure game enhancement?
  3. Modularity is Key: Strive to keep new features or fixes well-contained within the appropriate module or file.
  4. Follow Patterns: Observe existing command definitions in CommandExecutor.commands (in commexec2_0.js) for structure (handler, description, helpText). Note how they use utilities from oopisos2_0.js (like Utils.validateArguments, FileSystemManager.validatePath).
  5. User Experience: Ensure clear error messages via OutputManager. Use ConfirmationManager (from oopisos2_0.js) for terminal-based confirmations or EditorModal (from editor2_0.js) for confirmations within the editor context. For password entry, see the `PasswordPromptManager` in `oopisos2_0.js`.
  6. Persistence: Any changes to the file system must be saved using FileSystemManager.save(). Session or setting changes might use StorageManager or affect data saved by SessionManager.
  7. Testing:
    • Thoroughly test your changes. Consider various inputs and edge cases.
    • Leverage or extend existing test scripts (e.g., diag2_0.sh) or create new ones using the run command and check_fail for assertions.
  8. Documentation: Update the command's description and helpText within its definition in commexec2_0.js. If significant changes are made, consider updates to this HTML manual.

Developing New Commands

Adding a new command involves defining it within the commands object in the CommandExecutor module (located inside commexec2_0.js). Here's the basic structure:


// Inside commexec2_0.js, within CommandExecutor IIFE, in the 'commands' object:
"yournewcommand": {
    handler: async (args, options) => {
        // args: Array of string arguments from the parsed command line.
        // options: Object containing:
        //   - isInteractive: Boolean, true if command run directly in terminal.
        //   - stdinContent: String or null, content from a pipe if applicable.
        //   - explicitForce: Boolean, true if -f or --force was in args (convenience).

        // 1. Validate arguments (use Utils.validateArguments from oopisos2_0.js)
        //    Example: const valRes = Utils.validateArguments(args, { exact: 1 });
        //             if (!valRes.isValid) return { success: false, error: `mycmd: ${valRes.errorDetail}` };

        // 2. Parse flags if your command uses them (Utils.parseFlags from oopisos2_0.js)
        //    Example: const { flags, remainingArgs } = Utils.parseFlags(args, [{name: "verbose", short: "-v"}]);

        // 3. Implement your command's logic.
        //    You can interact with modules like FileSystemManager, UserManager, OutputManager, etc.,
        //    which are globally available from oopisos2_0.js.
        //    Example: const pathInfo = FileSystemManager.validatePath("mycmd", remainingArgs[0]);
        //             if (pathInfo.error) return { success: false, error: pathInfo.error };
        //             OutputManager.appendToOutput(`Processing: ${pathInfo.resolvedPath}`);

        // 4. Return a result object.
        //    - success: Boolean (true if successful, false otherwise).
        //    - output: String or null (text to display if successful and not redirected).
        //    - error: String or null (error message if success is false).
        //    - messageType: Optional string (CSS class from Config.CSS_CLASSES for output styling).
        
        return { success: true, output: `New command processed: ${args.join(' ')}`, messageType: Config.CSS_CLASSES.SUCCESS_MSG };
    },
    description: "A brief, one-line description of your new command for 'help'.",
    helpText: "Usage: yournewcommand [options] [arguments]\\n\\nProvide detailed help here, including options, arguments, and examples. Explain what the command does."
}

Refer to existing commands in commexec2_0.js for comprehensive examples of argument validation, flag parsing, interacting with the file system (including permission checks with FileSystemManager.hasPermission), handling piped input, and producing structured output.

Happy developing with OopisOS v2.0!

Testing & Showcase Environment

OopisOS v2.0 includes two important shell scripts, inflate2_0.sh and diag2_0.sh, designed to streamline testing and provide a rich user experience out of the box. You can execute them using the run command.

Showcase Data with `inflate2_0.sh`

The inflate2_0.sh script is a "Test Drive Population Script." Its purpose is to populate a fresh file system with a diverse and interesting set of files and directories. This creates a showcase environment where new users can immediately begin experimenting with commands like ls, grep, find, tree, `gemini`, and `adventure` without needing to create all the content themselves.

To use it, simply run:

Guest@OopisOs:/> run /inflate2_0.sh

What it Creates:

  • A root-level `README.md` explaining the test drive environment.
  • A `/docs` directory with sample documentation.
  • A `/src` directory with simulated source code files.
  • A `/data` directory with text files perfect for `grep` practice.
  • A `/reports` directory containing a sample financial report for use with the `gemini` command.
  • A `/games` directory with the `quest_for_the_semicolon.json` adventure file.
  • Hidden files (e.g., `.env`) for `ls -a` and `find` discovery.
  • Files with specific past timestamps for testing `touch` and `ls -t`.

This script makes it easy to demonstrate the full capabilities of OopisOS to new users or for developers to quickly set up a populated testbed.

Core Diagnostics with `diag2_0.sh`

The diag2_0.sh script is a non-interactive test suite designed to rigorously verify the core functionality of OopisOS commands. It's an essential tool for developers to ensure that changes to the codebase haven't introduced regressions.

This script automatically performs a sequence of operations, printing its actions and expected outcomes, and uses the special `check_fail` command to assert that error conditions are handled correctly.

Key Test Phases:

  • Workspace Setup: Logs in as the special `userDiag` user (password: `pantload`) and creates a clean `diag_workspace` directory.
  • File System Tests: Verifies `mkdir`, `touch`, `ls`, `pwd`, and `cd`, including error cases like trying to create a directory that already exists.
  • File Content Tests: Checks `echo`, `cat`, and I/O redirection (`>` and `>>`) and pipelining (`|`).
  • Advanced `ls` Flags: Tests all sorting and formatting flags of the `ls` command.
  • Permissions & Ownership: Runs a complex scenario involving `chmod`, `chown`, switching between `userDiag` and `Guest` users to verify that read/write/execute permissions are enforced correctly.
  • File Management: Tests `rm`, `cp`, and `mv`, including recursive and force operations.
  • `find` Command: Sets up a test directory structure and verifies `find` with various predicates (`-name`, `-type`, `-user`) and actions (`-exec`, `-delete`).
  • `run` Command: Tests the script runner itself with argument passing.
  • Cleanup: Switches to the `root` user to delete the diagnostic workspace and restores the system to a clean state.

Running diag2_0.sh is a great way to see a wide range of OopisOS features in action and to verify the health of the system.

Guest@OopisOs:/> run /diag2_0.sh
